# 进程和线程

TIP

进程（process）指计算机已经运行的程序，是操作系统管理程序的一种方式，我们可以认为，当我们启动一个应用程序时，默认至少启动了一个进程。

线程（thread）指操作系统能够运行运算调度的最小单位，一般情况下，它被包含在进程中。每一个进程中，都会至少启动一个线程来运行程序的代码，这个线程被称为主线程，所以通常说进程是线程的容器。

# 函数调用栈

在我们学习 JavaScript 的事件循环之前，我们需要先了解一下 JavaScript 中函数的执行过程。函数的执行过程其实是一个压栈的过程。我们可以把栈结构比喻成一个桶，我们往里面放东西，先放进去的东西被压在下面，如果想把它拿出来，必须把上面的东西全拿出来才能拿最下面的。因此栈结构最显著的特点就是先进后出。

console.log("1");

function add(num1, num2) {
  return num1 + num2;
}

function result() {
  const res = add(1, 2);
  console.log(res);
}

result();

// 这段代码的出入栈顺序是怎样的呢？我们一起来分析一下：

/* 假设一开始的函数调用栈是空的一个数组 */

let stack = [];

/* 1. 第一步执行了 console.log()函数，此时函数调用栈新增内容 */

stack = ["console.log()"];

/* 2. 执行完毕之后弹出函数调用栈，此时函数调用栈又成为了空的 */

stack = [];

/* 
  3. add和result函数进行声明，未执行，并不会进入到函数调用栈 
  4. 执行了result， 此时result函数进入函数调用栈
*/

stack = ["result"];

/* 5. 执行result的过程中，发现add函数进行执行，add函数进入到调用栈 */

stack = ["result", "add"];

/* 6. add函数先执行完毕，弹出函数调用栈，然后result函数也执行完毕，接着跳出函数调用栈，此时调用栈为空 */

stack = [];

# 任务队列

什么叫做队列呢？

我们可以简单理解为一个狭窄的隧道，一个人先从隧道进去，然后又进去一个人，如果前面的人不出去，后面的人也出不去。因此队列的特征就是先进先出。

众所周知，js 是一门单线程语言，因此同一时刻只能有一行代码在执行，代码的执行过程由上而下依次执行。因此当遇到一些耗时的操作，进程就会被阻塞，下面的代码都将不能执行，如何解决这种情况呢？JavaScript 将任务分为了同步任务以及异步任务。

TIP

  • 同步任务 指的是，所有任务都在一个进程中，只有执行完上一个任务，才可以执行下一个任务，同步任务一般在主线程执行。
  • 异步任务 指的是不进入主线程，而是通过调用 web api 先进入任务队列进行等待，然后在特定时机进入主线程开始执行。

任务队列并不是只有一个队列，它可以细分为宏任务队列(macrotask queue)和微任务队列(microtask queue)。

TIP

常见的宏任务有：定时器、ajax、UI 渲染、dom 监听等

常见的微任务有：Promise.then()、queueMicrotask()、Object.observe()、async/await 等

# 什么叫事件循环

好了，了解完函数调用栈以及任务队列，我们就要具体的来了解一下事件循环了，什么叫事件循环呢？

TIP

在 JavaScript 中，同步任务和异步任务分别进入不同的执行环境。同步任务进行主线程，异步任务进入任务队列，等到主线程的任务执行完毕后，会去任务队列取出合适时机的任务，将它放入主线程的函数调用栈中执行，这个过程就是事件循环。*

那么它具体的执行过程和顺序是怎样的呢？

  1. 执行微任务队列，将微任务队列中的任务执行完毕。

  2. 执行宏任务队列，执行完宏任务队列中的第一个任务。

  3. 查看微任务队列是否有新加入的微任务。

  4. 如果没有新增，继续执行宏任务的下一个任务，如果微任务队列有新增任务，继续执行微任务队列中的所有任务。

  5. 重复步骤 3 和步骤 4。

  6. 执行完毕任务队列中所有任务。

所以说，事件循环的优先级为 同步任务 > 微任务 > 宏任务

# 定时器以及 Promise 函数体的执行时机

定时器：

我们都知道，setTimeout以及setInterval本身就是一个函数，那么这些函数是否会影响代码的执行顺序呢？

其实是不会的，原因就是定时器函数执行时会被压入函数栈，进入函数栈后会立马执行，然后弹出函数栈。因此这些函数本身是不会对我们后续的任务造成阻塞。

而它们的回调函数会在合适的时机加入到宏任务队列，比如我们一个定时器的延时时间设置为 1000 毫秒，那么在1000 毫秒之后，它的回调函数就会被加入到宏任务队列。

Promise：

Promise 函数体内的代码，会被当作同步代码执行，但是Promise.then, Promise.catch，Promise.finally，Promise.resolve()，Promise.reject()会被加入到微任务队列中执行。

根据我们的规则，我们应该可以理解为什么 123 和 456 先打印，因为它们都是同步任务；

然后按照同步任务、微任务、宏任务的顺序依次执行。

setTimeout(() => {
  console.log("set1");
  new Promise((resolve) => {
    resolve();
  }).then(() => {
    new Promise((resolve) => {
      resolve();
    }).then(() => {

    console.log("then2");
  });

我们来看一下上述代码的执行过程：

  1. 执行 setTimeout 函数，回调函数(set1)放入宏任务队列
  2. 执行 new Promise，因为 promise 回调函数的主体会放入主进程执行，因此打印出promise1， 接下来执行 resolve()，进入 promise.then，promise.then 的回调(then1)被放入微任务队列
  3. 执行 setTimeout 函数，回调函数(set2)放入宏任务队列
  4. 执行 console.log 函数，进入主进程，打印出 log
  5. 自定义微任务，微任务(queueMicrotask)进入微任务队列
  6. 执行 new Promise，执行 resolve(),进入 promise.then，(then3)被放入微任务队列

至此，我们可以得到此时的情况：

已打印：promise1，log

微任务队列：then1,，queueMicrotask，then3

宏任务队列：set1，set2

继续执行微任务队列中的任务：

  1. 执行 console.log('then1')，放入函数调用栈打印出 then1
  2. 执行 console.log('queueMicrotask')，打印出 queueMicrotask
  3. 执行 console.log('then3')，打印出 then3
  4. 微任务执行完毕，开始执行宏任务 set1，首先执行 console.log('set1')，打印出 set1，紧接着执行 new Promise，执行 resolve，进入 promise.then，将 new Promise 放入微任务队列，此时微任务队列新增了任务，因此会继续执行 new Promise，再次调用 resolve，将 then4 加入到微任务队列，此时 then2 在主线程执行，因此会打印出 then2。回头继续执行微任务，打印出 then4，微任务队列再次清空，开始执行宏任务。
  5. 执行 console.log('set2')打印出 set2
  6. 任务队列执行完毕，代码执行完成

从上述过程可得知最后的结果为 promise1 => log => then1 => queueMicrotask => then3 => set1 => then2 => then4 => set2
