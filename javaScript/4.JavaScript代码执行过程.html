# JavaScript 代码执行过程

# 全局

TIP

  • 在 js 代码被解析时，V8 引擎内部会帮助我们创建一个对象 GlobalObject，也就是我们常说的GO

  • GO里面会包含一些全局的变量或方法，并且有一个名为window的属性，而 window 的 value 指向 GO 本身，因此当我们访问 window.xx时，其实访问的是 GlobalObject 内部属性。

这也就是说，在 v8 引擎解析我们的 js 代码时，其实已经帮助我们创建了一个全局对象 globalObject，该对象里面有全局的一些函数，比如我们常用的console、alert等。并且内部还有一个 window 对象，这个对象的值就是 globalObject。用代码的形式表示如下：

var globalObject = {
  console: ...,
  alert: function() {...},
  window: globalObject
}

这也就是我们平时在使用某些全局函数的时候，使用 window.xx 和直接使用 xx 效果是一样的。

TIP

  • V8 为了执行代码，内部会有一个执行上下文栈(Execution Context Stack)（函数调用栈）
  • V8 引擎为了保证全局代码的执行，在全局代码执行时会创建全局执行上下文（Global Execution Context），然后将它放入执行上下文栈进行执行
  • 全局执行上下文中会有一个变量对象（variable object），也可以称为VO，在全局执行上下文中，VO其实指向的是GO

# 函数

  • v8 引擎在解析函数时，会开辟一块新的内存地址，专门用来储存函数空间，里面存储了该函数的作用域[[scope]]，并且指向它父级的 scope，还保存了该函数的执行体。

TIP

  • 函数在执行时，会生成一个函数执行上下文（Functional Execution Context）
  • 函数执行上下文中也有VO，但是与变量不同的是，此时的 VO指向AO，AO 是一个活跃对象（Activation Object）
  • 当函数执行完毕之后，该函数的函数执行上下文会被销毁，此时如果没有东西指向现在的AO，AO也会被销毁
  • 在函数中，参数也会被添加到 VO 中

# 变量提升及全局代码执行分析

看完了上面的一些概念，是不是感到凌乱了，这啥啊都是？什么 AO、VO、GO 的❓

通过代码来看也许会更加清晰明了一点。我们先来看一个全局代码块的：

console.log(a); // undefined

var a = 123;

我们都知道，js 是单线程的，也就是说一次只能执行一行代码，按照我们理解的正常逻辑，在我们未声明变量 a的时候去打印 a，应该会抛出异常的，为什么会打印出 undefined 呢？

这是因为 v8 引擎在解析 js 代码的时候，会帮助我们把声明变量的操作提前，这是什么意思呢？

首先在执行上述代码时，v8 引擎帮助我们创建了一个全局对象 GO：

var globalObject = {
  ...
}

看完上面的代码我们就可以清晰的发现，v8 引擎帮助我们做了什么，为什么我们先执行打印操作再声明变量，却还能打印出undefined，就是因为内部帮我们转换了一步，让我们的声明操作提前了，这样我们在执行打印的时候，其实已经声明了变量 a，只是还未赋值，因此不会抛出异常，只是会打印出 undefined。

TIP

而这种内部将变量声明提前的情况，我们就可以叫它变量提升。

从上面的代码我们可以知道，当一个普通变量先打印再声明，因为变量提升的存在我们会打印出 undefined，要是一个特殊的变量呢？比如一个函数？它也会打印出 undefined 吗？上代码看一下：

console.log(foo); // f foo() { ... }
console.log(bar); // undefined

function foo() {
  console.log("社交名流");
}

var bar = function () {
  console.log("社交名流");
};

执行上述代码我们可以发现，情况又不一样了！foo 函数可以在声明之前就可以打印出来，而 bar 函数却还是 undefined，这是为啥呢？

这是因为当我们解析一个函数时，其实会开辟一块新的内存地址，这也就意味着在解析 foo 函数时，假设 foo 开辟的内存地址是 0x100，那么在解析完毕后，GO 中 foo 就是这样的存在方式：

var globalObject = {
  foo: 0x100,
  ...
}

var globalObject = {
  foo: 0x100,
  bar: undefined,
  ...
}

上述代码会打印出什么呢？我们一起分析一波：

// 1. 解析
// 我们知道函数在执行时会生成函数执行上下文，而这个函数执行上下文其实就是AO对象，所以在解析foo函数时：
var activationObject = {};
// 函数内的VO指向AO，即：
var variableObject = activationObject;
// 函数的参数会被添加到VO中，函数体内声明的变量也会添加到VO中，如果参数和变量名一样呢？我们只取一个，即：
variableObject.a = undefined;
variableObject.b = undefined;
// 2. 执行
// 执行foo函数，传入实参1，2，此时实参覆盖解析时的值，即：
variableObject.a = 1;
variableObject.b = 2;
// 3. 打印变量a
console.log(a); // return 1
// 4. 重新赋值变量a
a = 10;
// 5. 重新打印变量a
console.log(a); // return 10
// 6. 打印变量b
console.log(b); // return 2
// 7. 重新赋值变量b
b = 20;
// 8. 重新打印变量b
console.log(b); // return 20

有人会说，诶？你代码写的 var a = 10，var b = 20，为啥你分析的时候不要 var 了？

这是因为声明变量那一步在解析时已经进行了，当 variableObject 对象给 a、b 赋值 undefined 的时候就已经执行了 var a 和 var b 了，下面的操作都是赋值操作罢了。

# 整体梳理

我们将 AO、VO、GO 放在一起梳理一下。

console.log(a); // undefined
var a = 1;

foo(); // undefined, foo
function foo() {
  console.log(i);
  var i = 1;
  var j = 2;
  console.log("foo");
}

我们一起看一下这段代码，这段代码是怎么执行的呢？首先它们整体肯定是在全局进行执行的，那么我们首先来分析一些全局执行的流程。

// 1. 首先在全局执行上下文会生成一个GO和VO，并且VO和GO指向同一块内存地址
// 2. 变量提升，因为foo为函数，会开辟新的内存地址，因此此时全局GO/VO为这样
var go = {
  a: undefined,
  foo: "内存地址",
};
// 3. 执行代码
console.log(a); // return undefined
// 4. 对变量a进行赋值
a = 1;
// 5. `()`为执行语句，执行foo函数
foo();
// 6. 创建函数执行上下文，函数执行上下文创建A0/VO，此时函数执行上下文的A0/VO为：
var ao = {
  i: undefined,
  j: undefined,
};
// 7. 执行函数体
console.log(i); // return undefined
// 8. 对变量进行赋值操作
i = 1;
j = 2;
// 9. 打印
console.log("foo"); // return foo

所以说无论当我们碰见什么样的情况，我们只需要一步一步分析出全局执行上下文的 GO/VO以及函数执行上下文的 A0/VO，就可以对它们的执行顺序一目了然。

TIP

在最新的 ECMA 标准中，官方把Variable Object改为了Variable Environment（变量环境），环境中定义的一些变量或函数将会作为一条环境记录（environment record）绑定在变量环境中。 之所以更改，是因为VO 不一定非要是一个对象，也可以是Map等数据结构来存储变量，更加严谨。
